# ASCF 逻辑层服务架构分析

## 一、整体定位

逻辑层是 ASCF 框架的 JavaScript 运行时核心，主要负责为小程序 / 快应用提供完整的逻辑运行环境。

它承担的核心职责包括：

* 框架初始化
* 应用启动
* App 生命周期管理
* Page 页面生命周期管理
* Component 组件系统
* Behavior 行为复用机制
* JSBridge 通信
* API 模块管理
* API 注册与分发
* 逻辑层与原生层、视图层之间的协作

可以简单理解为：

```text
逻辑层 = 小程序运行时的大脑
```

它不直接负责页面最终渲染，也不直接实现所有鸿蒙原生能力，而是负责组织 JavaScript 逻辑、管理生命周期、调用 API，并通过桥接层与原生层和视图层通信。

## 三、核心功能模块

## 1. 框架初始化

### 入口文件

```text
index.js
```

### 主要职责

框架初始化模块是逻辑层的入口，主要负责启动整个 JavaScript 运行环境。

它的主要职责包括：

1. 初始化 WebApp 框架环境。
2. 注册全局对象：

    * `App`
    * `Page`
    * `Component`
    * `Behavior`
3. 订阅 JSBridge 事件。
4. 建立与原生层的通信通道。
5. 准备后续 app、page、component 的注册和运行环境。

可以理解为：

```text
index.js
  ↓
初始化逻辑层运行环境
  ↓
注册小程序全局构造方法
  ↓
建立 JSBridge 通信
  ↓
等待加载 app.js 和页面逻辑
```

---

## 2. 应用启动器

### 模块位置

```text
service/appLauncher.js
```

### 主要职责

应用启动器负责启动小程序应用逻辑，是从框架初始化进入业务 app 逻辑的重要环节。

它的主要职责包括：

1. 加载并执行小程序的 `app.js`。
2. 解析 `app.json` 配置。
3. 初始化应用启动流程。
4. 触发应用级生命周期。
5. 将应用配置、页面配置和运行时环境串联起来。

可以理解为：

```text
框架初始化完成
  ↓
应用启动器开始工作
  ↓
加载 app.json
  ↓
执行 app.js
  ↓
创建 App 实例
  ↓
进入应用生命周期
```

---

## 3. 应用管理器

### 模块位置

```text
service/appManager.js
```

### 主要职责

应用管理器负责创建和维护 App 实例，是应用级状态和生命周期的管理中心。

它的主要职责包括：

1. 创建和管理 App 实例。
2. 绑定应用生命周期钩子。
3. 管理应用全局状态。
4. 处理应用级别的事件。
5. 统一维护应用运行过程中的上下文信息。

### 应用生命周期

| 生命周期       | 触发时机            |
| ---------- | --------------- |
| `onLaunch` | 应用初始化完成时触发      |
| `onShow`   | 应用启动或从后台进入前台时触发 |
| `onHide`   | 应用从前台进入后台时触发    |
| `onError`  | 应用发生脚本错误时触发     |

### 理解方式

```text
App({
  onLaunch() {},
  onShow() {},
  onHide() {},
  onError() {}
})
```

这些生命周期最终需要被应用管理器识别、保存，并在合适的时机触发。

---

## 4. 页面管理器

### 模块位置

```text
service/PageManager.js
```

### 主要职责

页面管理器负责管理小程序页面，是页面栈、页面路由和页面生命周期的核心管理模块。

它的主要职责包括：

1. 管理页面栈。
2. 支持页面导航。
3. 处理页面路由事件。
4. 管理页面生命周期。
5. 支持异步组件加载。
6. 处理页面参数传递。
7. 协调页面与视图层之间的数据更新。

### 页面栈管理

页面管理器通常需要维护页面历史记录，并支持常见路由行为，例如：

| 路由行为           | 作用              |
| -------------- | --------------- |
| `navigateTo`   | 保留当前页面，跳转到新页面   |
| `redirectTo`   | 关闭当前页面，跳转到新页面   |
| `navigateBack` | 返回上一个或多级页面      |
| `switchTab`    | 切换 tabBar 页面    |
| `reLaunch`     | 关闭所有页面，重新打开某个页面 |

### 页面生命周期

| 生命周期       | 触发时机        |
| ---------- | ----------- |
| `onLoad`   | 页面加载时触发     |
| `onShow`   | 页面显示时触发     |
| `onReady`  | 页面初次渲染完成时触发 |
| `onHide`   | 页面隐藏时触发     |
| `onUnload` | 页面卸载时触发     |

### 理解方式

```text
页面跳转
  ↓
页面管理器更新页面栈
  ↓
创建或复用页面实例
  ↓
触发页面生命周期
  ↓
通知视图层渲染页面
```

## 5. 组件系统

### 模块位置

```text
service/Component.js
```

### 主要职责

组件系统负责定义和管理小程序组件，是页面内部组件化能力的基础。

它的主要职责包括：

1. 定义组件基类。
2. 管理组件生命周期。
3. 处理组件数据绑定。
4. 管理组件属性。
5. 管理组件事件。
6. 支持组件方法调用。
7. 协调组件与页面之间的数据关系。

### 组件生命周期

| 生命周期       | 触发时机       |
| ---------- | ---------- |
| `created`  | 组件实例化时触发   |
| `attached` | 组件挂载到页面时触发 |
| `ready`    | 组件渲染完成时触发  |
| `moved`    | 组件位置移动时触发  |
| `detached` | 组件卸载时触发    |

### 理解方式

```text
Component({
  properties: {},
  data: {},
  methods: {},
  lifetimes: {
    created() {},
    attached() {},
    ready() {},
    detached() {}
  }
})
```

逻辑层需要把这些配置转化为真实的组件实例，并在对应时机触发生命周期。

---

## 6. 行为管理器

### 模块位置

```text
service/BehaviorManager.js
```

### 主要职责

行为管理器负责实现小程序中的 Behavior 特性。

Behavior 可以理解为组件之间复用逻辑的一种方式，类似于组件 mixin。

它的主要职责包括：

1. 实现 Behavior 特性。
2. 支持 Behavior 混入。
3. 管理 Behavior 生命周期。
4. 合并 Behavior 中的数据。
5. 合并 Behavior 中的方法。
6. 处理组件自身配置和 Behavior 配置之间的优先级。

### 理解方式

```text
Behavior
  ↓
被多个 Component 引用
  ↓
合并 data / methods / lifetimes
  ↓
生成最终组件配置
```

Behavior 管理器的重点不是渲染，而是处理逻辑复用和配置合并。

---

## 7. JSBridge 桥接层

### 模块位置

```text
service/bridge/jsBridge.js
```

### 主要职责

JSBridge 桥接层负责实现 JavaScript 与原生层之间的通信，是逻辑层调用鸿蒙能力的关键通道。

它的主要职责包括：

1. 实现 JS 与原生层通信。
2. 实现异步调用机制。
3. 管理回调函数。
4. 处理跨语言数据转换。
5. 维护请求与响应之间的对应关系。
6. 接收原生层事件通知。
7. 将 API 调用转发给原生能力实现。

### 通信机制

JSBridge 通常需要处理以下问题：

1. 消息队列。
2. 同步调用。
3. 异步调用。
4. 请求 ID 管理。
5. 回调函数缓存。
6. 数据序列化。
7. 数据反序列化。
8. 错误返回。
9. 超时或异常处理。

### 理解方式

```text
JS 调用 API
  ↓
JSBridge 生成请求
  ↓
发送到原生层
  ↓
原生层执行能力
  ↓
返回结果
  ↓
JSBridge 找到对应回调
  ↓
触发 success / fail / complete
```

## 8. 模块管理

### 模块定位

模块管理主要负责提供各类 API 的定义和部分实现，用来兼容微信小程序 API，降低小程序代码迁移成本。

可以理解为：

```text
模块管理 = wx.xxx API 的组织中心
```

例如：

```text
wx.getSystemInfo()
wx.request()
wx.showToast()
wx.setStorage()
```

这些 API 通常会被组织到不同能力模块中。

### API 模块结构

```text
module/wx
├── api                       # API 模块
├── base                      # 基础模块
├── device                    # 设备模块
├── media                     # 媒体模块
├── network                   # 网络模块
├── service                   # 服务模块
├── ui                        # UI 模块
├── AI                        # AI 能力模块
├── interface.js              # API 统一导出入口
├── nativeComponentEvent.js   # 原生组件事件
└── utils                     # 工具模块
```

### 主要职责

1. 按能力类型组织 API。
2. 兼容微信小程序 API 形态。
3. 提供统一导出入口。
4. 管理原生组件事件。
5. 为 API 注册模块提供可注册的接口。
6. 屏蔽底层原生能力差异。

---

## 9. API 注册

### 模块位置

```text
service/api/register.js
```

### 主要职责

API 注册模块负责统一注册逻辑层可用的 API。

它的主要职责包括：

1. 注册所有可用 API。
2. 建立 API 名称与实现之间的映射。
3. 提供 API 查询接口。
4. 判断某个 API 是否存在。
5. 为 JSBridge 调用提供分发基础。
6. 为兼容小程序 API 提供统一注册表。

### 理解方式

```text
API 实现文件
  ↓
统一导出入口
  ↓
API 注册模块
  ↓
生成 API 映射表
  ↓
JS 调用时根据 API 名称查找实现
```

如果出现“API 不存在”或“API 调用没有进入实现”的问题，通常需要优先检查 API 是否完成注册。

## 10. 与其他模块的交互

逻辑层不是孤立运行的，它需要同时和底层核心层、原生层、视图层交互。

## 10.1 与底层核心层 / 原生层交互

逻辑层主要通过 JSBridge 与原生层通信。

交互内容包括：

1. 调用原生能力。
2. 接收原生层事件通知。
3. 获取运行时配置。
4. 获取系统能力结果。
5. 向原生层发送页面或 API 请求。

链路可以理解为：

```text
逻辑层
  ↓
JSBridge
  ↓
原生桥接
  ↓
底层核心能力
  ↓
鸿蒙系统能力
```

## 10.2 与视图层交互

逻辑层与视图层之间主要围绕页面数据、事件和生命周期协作。

交互内容包括：

1. 提供页面数据给视图层渲染。
2. 接收视图层用户交互事件。
3. 协调页面生命周期。
4. 协调组件生命周期。
5. 通知视图层进行页面更新。
6. 接收视图层渲染完成的反馈。

链路可以理解为：

```text
视图层用户点击
  ↓
事件传递到逻辑层
  ↓
逻辑层执行事件处理函数
  ↓
更新页面数据
  ↓
通知视图层重新渲染
```

## 11. 新增 API 的大致流程

如果后续需要添加一个新的 API，可以先按下面的流程理解。

### 11.1 新增 API 文件

在对应能力分类下创建新的 API 文件。

例如：

```text
基础能力
设备能力
媒体能力
网络能力
存储能力
UI 能力
开放能力
```

### 11.2 实现 API 逻辑

在 API 文件中实现具体功能逻辑。

通常需要考虑：

1. API 名称。
2. 入参校验。
3. 默认参数。
4. success 回调。
5. fail 回调。
6. complete 回调。
7. 是否需要调用原生能力。
8. 是否需要返回 Promise。
9. 错误码和错误信息。
10. 是否兼容小程序 API 行为。

### 11.3 导出 API

在统一导出入口中导入新的 API。

### 11.4 注册 API

将新的 API 添加到 API 注册列表或接口映射对象中。

### 11.5 通过 JSBridge 调用原生能力

如果该 API 需要调用鸿蒙系统能力，则需要通过 JSBridge 或原生桥接层继续向下分发。

### 11.6 验证 API

需要验证以下内容：

1. API 是否能被 JS 侧访问。
2. 参数是否正确传递。
3. 原生能力是否被调用。
4. success / fail / complete 是否按预期触发。
5. 异常场景是否有返回。
6. 和小程序 API 行为是否一致。
7. 是否影响已有 API。

整体流程可以理解为：

```text
新增 API 文件
  ↓
实现 API 逻辑
  ↓
统一导出
  ↓
加入 API 注册表
  ↓
必要时接入 JSBridge
  ↓
调用原生能力
  ↓
补充测试和验证
```

## 十二、逻辑层主运行链路

综合上面的模块，可以把逻辑层运行流程理解为：

```text
框架入口启动
  ↓
初始化逻辑层运行环境
  ↓
注册 App / Page / Component / Behavior
  ↓
建立 JSBridge 通信
  ↓
应用启动器加载 app.json
  ↓
应用启动器执行 app.js
  ↓
应用管理器创建 App 实例
  ↓
触发 App 生命周期
  ↓
页面管理器创建 Page 实例
  ↓
触发 Page 生命周期
  ↓
组件系统创建 Component 实例
  ↓
Behavior 管理器合并复用逻辑
  ↓
API 注册模块注册可用 API
  ↓
模块管理提供 wx 兼容 API
  ↓
JSBridge 负责和原生层通信
  ↓
视图层接收数据并渲染页面
```

## 十三、问题排查方向

| 问题现象                          | 优先排查方向              |
| ----------------------------- | ------------------- |
| 框架没有启动                        | 入口初始化、全局对象注册        |
| `App` 不存在                     | 框架入口、全局对象注册         |
| `Page` 不存在                    | 框架入口、页面注册流程         |
| `Component` 不存在               | 框架入口、组件系统           |
| `app.js` 没有执行                 | 应用启动器               |
| `app.json` 配置不生效              | 应用启动器、配置解析          |
| `onLaunch` 没触发                | 应用管理器、生命周期绑定        |
| `onShow` / `onHide` 异常        | 应用管理器、前后台状态         |
| 页面无法跳转                        | 页面管理器、页面栈、路由事件      |
| `onLoad` 没触发                  | 页面管理器、页面创建流程        |
| 页面参数丢失                        | 页面管理器、路由参数传递        |
| 组件生命周期异常                      | 组件系统、Behavior 合并    |
| Behavior 不生效                  | 行为管理器               |
| `wx.xxx` API 不存在              | 模块管理、API 注册         |
| API 调用无返回                     | JSBridge、回调管理器      |
| success / fail / complete 不触发 | 回调管理器、JSBridge 响应分发 |
| 原生事件没有收到                      | JSBridge 事件订阅       |
| 视图层点击无响应                      | 视图层事件传递、逻辑层事件处理     |
| 数据更新页面不刷新                     | 逻辑层数据更新、视图层渲染通知     |

## 十四、当前理解总结

逻辑层可以总结为以下几个关键词：

```text
入口初始化
应用启动
生命周期管理
页面管理
组件系统
Behavior 复用
JSBridge 通信
API 模块管理
API 注册分发
微信小程序兼容
逻辑层与视图层协作
逻辑层与原生层协作
```

更简洁地说：

```text
逻辑层负责让小程序代码真正跑起来。
```

它向上承接小程序开发者写的 `app.js`、页面 JS、组件 JS，向下通过 JSBridge 调用鸿蒙原生能力，同时把数据和事件同步给视图层完成页面渲染。
