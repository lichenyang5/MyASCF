# 第一阶段验收记录

## Step 1：ArkWeb 加载 H5 页面

目标：
验证 ArkTS Web 组件能加载 rawfile/h5/index.html。

当前不做：
- JavaScriptProxy
- runJavaScript
- Controller
- Dispatcher
- Register
- Biz / Imp

验收标准：
1. 应用启动后能看到 ASCF Runtime Lab 页面。
2. 页面显示 H5 内容。
3. DevEco 日志中能看到 onPageBegin / onPageEnd。
4. 不出现白屏或资源加载失败。

## Step 2：JavaScriptProxy 注入 ascfBridge

目标：
验证 H5 能通过 window.ascfBridge.send 调用 ArkTS 方法。

当前调用链路：
H5 Button
  ↓
window.ascfBridge.send(rawReq)
  ↓
JavaScriptProxy
  ↓
AscfBridgeObject.send(rawReq)
  ↓
return response string
  ↓
H5 页面展示 response

验收标准：
1. 页面能看到 “Step 2：调用 ascfBridge.send” 按钮。
2. 点击按钮后，DevEco 日志能看到 ArkTS 收到 rawReq。
3. H5 页面能展示 ArkTS 返回的 response。
4. response 中包含 requestId、code、message、data。
5. 当前阶段不使用 runJavaScript。

## Step 3：runJavaScript 回调 H5

目标：
验证 ArkTS 能通过 WebviewController.runJavaScript 主动调用 H5 的 window.__ascfResolve(response)。

当前调用链路：

H5 Button
  ↓
window.ascf.send(...)
  ↓
window.ascfBridge.send(rawReq)
  ↓
JavaScriptProxy
  ↓
AscfBridgeObject.send(rawReq)
  ↓
ArkTS 构造 BridgeResponse
  ↓
WebviewController.runJavaScript(...)
  ↓
window.__ascfResolve(response)
  ↓
Promise resolve / reject
  ↓
H5 页面展示结果

本阶段对应 ASCF 架构图里的箭头：
1. H5 → ArkTS：JavaScriptProxy
2. ArkTS → H5：runJavaScript

验收标准：
1. 页面能看到 “Step 3：Promise 调用 ascfBridge.send” 按钮。
2. 点击按钮后，DevEco 日志能看到 ArkTS 收到 rawReq。
3. DevEco 日志能看到 runJavaScript 被调用。
4. H5 页面能看到 window.__ascfResolve 被调用。
5. H5 Promise 能 resolve。
6. 页面最终展示 response。
7. 当前阶段不实现 Dispatcher / Register / Biz / Imp。

## Step 4：BridgeRequest / BridgeResponse / BridgeController

目标：
把 H5 传来的 raw JSON 请求抽象成统一 BridgeRequest，并由 BridgeController 统一处理，再返回 BridgeResponse。

当前调用链路：

H5 Button
  ↓
window.ascf.send(action, params)
  ↓
BridgeRequest
  ↓
window.ascfBridge.send(rawReq)
  ↓
JavaScriptProxy
  ↓
AscfBridgeObject.send(rawReq)
  ↓
BridgeController.handle(rawReq)
  ↓
parseBridgeRequest(rawReq)
  ↓
successResponse / errorResponse
  ↓
WebviewController.runJavaScript(...)
  ↓
window.__ascfResolve(response)
  ↓
Promise resolve / reject

本阶段新增概念：
1. BridgeRequest：H5 到 ArkTS 的统一请求格式。
2. BridgeResponse：ArkTS 到 H5 的统一响应格式。
3. BridgeController：桥接控制层，负责解析请求、校验协议、组装响应。
4. AscfBridgeObject：只作为 JavaScriptProxy 暴露给 H5 的入口，不直接处理业务。

验收标准：
1. 页面能看到 “Step 4：通过 BridgeController 处理请求” 按钮。
2. 点击后 H5 能 Promise resolve。
3. DevEco 日志中能看到 [ASCF][Bridge]。
4. DevEco 日志中能看到 [ASCF][Controller]。
5. response 中包含 requestId、code、message、data。
6. data.stage 等于 BridgeController。
7. 当前阶段没有 Dispatcher / Register / Biz / Imp。

## Step 5：Dispatcher / Register / Handler

目标：
把 request.action 从一个普通字符串，变成可以被分发执行的能力标识。

当前调用链路：

H5 Button
  ↓
window.ascf.send(action, params)
  ↓
BridgeRequest
  ↓
window.ascfBridge.send(rawReq)
  ↓
JavaScriptProxy
  ↓
AscfBridgeObject.send(rawReq)
  ↓
BridgeController.handle(rawReq)
  ↓
parseBridgeRequest(rawReq)
  ↓
BridgeDispatcher.dispatch(request)
  ↓
HandlerRegister.getHandler(action)
  ↓
BridgeHandler.handle(request)
  ↓
BridgeResponse
  ↓
WebviewController.runJavaScript(...)
  ↓
window.__ascfResolve(response)
  ↓
Promise resolve / reject

本阶段新增概念：
1. BridgeDispatcher：分发器，只根据 action 找 handler，不写业务。
2. HandlerRegister：注册表，维护 action 到 handler 的映射。
3. BridgeHandler：具体 action 的处理函数。
4. UNKNOWN_ACTION：当 action 没有注册时，返回 404。

本阶段对应 ASCF 架构图里的箭头：
Controller
  ↓
Dispatcher
  ↓
Register
  ↓
Handler

验收标准：
1. 页面能看到 Step 5 的三个按钮。
2. runtime.ping 能返回 code: 0。
3. runtime.echo 能把 params 原样返回。
4. unknown.action 能返回 code: 404。
5. DevEco 日志中能看到 [ASCF][Dispatcher]。
6. DevEco 日志中能看到 [ASCF][Register]。
7. 当前阶段不实现 Biz / Imp，不调用 OpenHarmony API。

## Step 6：Biz / Imp 分层

目标：
把 action handler 拆成 Biz / Imp 两层，让 Register 不再直接写业务逻辑。

当前调用链路：

H5 Button
  ↓
window.ascf.send(action, params)
  ↓
JavaScriptProxy
  ↓
AscfBridgeObject
  ↓
BridgeController
  ↓
BridgeDispatcher
  ↓
HandlerRegister
  ↓
Biz
  ↓
Imp
  ↓
BridgeResponse
  ↓
runJavaScript
  ↓
H5 Promise resolve

本阶段新增概念：

1. Biz：
   业务封装层，负责解释 action 的业务语义。
   例如 runtime.ping、device.getInfo。

2. Imp：
   能力实现层，负责真正执行能力。
   当前阶段先返回 mock 数据，下一阶段再接 OpenHarmony API。

3. Register：
   只维护 action 到 handler 的映射，不直接写业务。

本阶段对应 ASCF 架构图里的箭头：

Register
  ↓
Biz
  ↓
Imp

验收标准：
1. 页面能看到 Step 6 的三个按钮。
2. runtime.ping 能经过 RuntimeBiz -> RuntimeImp。
3. runtime.echo 能经过 RuntimeBiz -> RuntimeImp。
4. device.getInfo 能经过 DeviceBiz -> DeviceImp。
5. DevEco 日志中能看到 [ASCF][RuntimeBiz]。
6. DevEco 日志中能看到 [ASCF][RuntimeImp]。
7. DevEco 日志中能看到 [ASCF][DeviceBiz]。
8. DevEco 日志中能看到 [ASCF][DeviceImp]。
9. 当前阶段 device.getInfo 返回 mock 数据，不调用真实 OpenHarmony API。
